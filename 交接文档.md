## 交接总述

## 一、论文的故事主线

### 1. 研究背景

低空交通/UAM 正在从零散示范转向网络化建设。城市侧一边在布局起降场，一边在讨论固定航线、可复用航路和走廊网络。问题在于，终端接入逻辑并没有收敛到单一模板。现实系统中，完全可能同时出现 procedure-based 的 legacy terminals 和 path-based 的新型 vertiports。于是，网络设计者面对的对象不再是一组同质节点，而是一组接入逻辑、局部几何和近端空域占用都不相同的终端。

### 2. 文献断层

现有研究大致分成三块。
第一块是**外部航路网络**，核心任务是设计城市级可复用航路、走廊和流量组织，但通常把终端接口视为外生固定。
第二块是**终端区容量与调度抽象**，说明终端区可以被系统级压缩成容量—延误对象。
第三块是**终端区运行构型与规则**，说明等待环、多环、合流/分流控制等内部设计会显著改变终端行为。

断层恰好出现在这三块之间：外部网络文献需要一个可调用的终端接口，终端研究却没有把内部构型系统性地压成一个供网络设计直接使用的中层表示。这篇论文就是去补这层。

### 3. 核心矛盾

网络设计想把终端压得越粗越好，方便算、方便分层；终端异质性又确实会穿透到网络层，压得过粗会选错连边、错估拥堵、错判近端冲突。
所以这篇论文真正关心的是：**为了把低空可复用航路网络设计做对，终端区最少要向上层暴露什么信息。**
这个问题一旦回答清楚，工程上就有机会重新回到分阶段设计。

### 4. 论文的中心命题

论文的中心命题可以表述为：

> 终端异质性并不需要整体抬升到上层网络优化，但也不能完全被压成普通节点。关键在于识别一个足够细、又足够轻的 network-facing interface，使得网络决策基本保持正确。

这也是压缩文档里“最小充分抽象”“决策等价优先于输出等价”“先联合验证接口，再恢复分层”的主线。

### 5. 论文的立场

论文的立场已经固定：

* 联合建模是研究工具，用来识别接口需求；
* 真正的研究交付是一个**decision-preserving interface**；
* 一旦接口成立，工程上应恢复分阶段设计，而不是永久维持大一统联合模型。

### 6. 终端为什么会影响网络

当前已经把终端穿透到网络层的主通道压成三类：

* **A：Access admissibility**
  改变“哪些 connector / 方向 / 端口可以接”。

* **S：Service capability**
  改变“接上以后 terminal 端的服务能力、延误与拥堵”。

* **F：Footprint / conflict**
  改变“terminal 邻域会不会挡住、惩罚或重排附近走廊”。

扩展层还有 **C：compatibility / contingency**，用于资格、关闭、退化和应急。第一篇主文先围绕 **ASF**。

### 7. 论文的方法逻辑

论文的方法逻辑已经固定成一条清楚的链：

**raw terminal factors → canonical factor families → intermediate states → ASF → network design**

具体地：

* 先把大量原始终端因素归并为 (P,G,R,O,X,[C]) 六个 canonical families；
* 再通过终端中观模型，把这些 family 映射到 (H,Q,B,[C]) 三类中间状态；
* 再从中间状态抽取网络层真正可见的接口 (A,S,F,[C])；
* 最后把这些接口嵌入到完整候选网络的设计问题里，比较不同抽象层级会不会误导航路网络。

### 8. 论文要回答的三个问题

整篇文章最后落到三个问题：

1. 把终端当成普通节点，会不会误导航路网络设计？
2. 终端异质性主要通过哪些通道影响上层决策？
3. 在什么条件下，S-only 足够，什么时候必须 AS，什么时候 ASF 才够？

### 9. 论文的最终贡献

当前设定下，论文希望交付四点：

1. 一个面向网络优化的终端接口框架；
2. 一个把终端设计空间压成网络可见接口的归并逻辑；
3. 一个从接口抽象误差到网络决策误差的验证框架；
4. 一套关于“何时可重新分层设计”的 regime 结论。

## 交接摘要

### 1. 论文定位

这篇论文的定位已经固定为：**低空可复用航路网络设计中的非点终端问题**。
研究对象是战略层面的航路/走廊网络设计；终端区进入上层时，不再被当成“一个点 + 一个总容量”，而是作为会改变连边可行性、端口服务能力和邻域冲突的边界对象进入网络优化。联合视角的用途，是识别上层设计真正需要的接口信息；工程目标是在接口足够后恢复分阶段设计。

### 2. 已冻结的核心问题

论文回答三个问题：

1. 把终端当成普通节点，会不会误导航路网络设计？
2. 终端异质性主要通过哪些 network-facing 通道穿透到上层？
3. 什么接口层级足以保持网络决策基本正确？

当前已经把接口固定为：

* **A**：access admissibility，终端允许哪些 connector / 方向 / port 接入
* **S**：service capability，端口级服务能力与延误
* **F**：footprint/conflict，终端邻域对外部边的阻断或惩罚

扩展层还有 **C**：compatibility / contingency，但第一篇主文先围绕 **ASF** 展开。

### 3. 当前的方法主线

终端内部原始因素很多，当前已经决定不用逐条硬塞进上层，而是走这条规范化链：

**raw factors → canonical factor families → intermediate states → ASF → network design**

具体写成：

* factor families：(P,G,R,O,X,[C])

  * (P)：procedure/interface regime
  * (G)：access-path geometry
  * (R)：resource configuration
  * (O)：local organization and control
  * (X)：operating context / surrounding airspace
  * (C)：compatibility / contingency（扩展）

* intermediate states：

  * (H)：interface geometry state
  * (Q)：service / queue state
  * (B)：boundary / footprint state

* network-facing interface：

  * (K_t=(A_t,S_t,F_t))

这层“factor → channel”逻辑是论文里必须明说的建模基础，正文里要有归并原则、一张 factor–channel matrix 和从 (u_t) 到 (K_t) 的代数定义；附录再放 long-list inventory 和编码规则。

### 4. 待比较的四个模型

实验统一比较四个层级：

* **M0：S-only**
  终端仍是粗节点，只保留聚合服务
* **M1：AS**
  终端展开为 ports，保留 admissibility 和 port-level service
* **M2：ASF**
  在 M1 基础上加入 footprint/conflict
* **M***：truth benchmark
  使用更细的 terminal truth model，只做 benchmark

论文的分层主线仍然对应原来的 A0 / A1 / A2：
A0 ≈ S-only，A1 ≈ AS，A2 ≈ ASF。

### 5. 上层网络模型

当前安排已经固定：

* 上层必须在**完整候选网络**上做设计，关心的是 topology、连通性、连边与流量；
* 下层 truth model 只在 terminal 邻域运行，不做全城逐机 4D 仿真；
* 网络图采用 **port-augmented candidate graph**；
* terminal 通过 ports 和 connectors 接到 backbone / corridor graph；
* 第一篇不以“新候选图生成算法”为贡献，候选图用固定图族即可；
* 求解层优先用现成的 MIQP / 分段线性 MILP / MCNF 风格框架，再把 ASF 接口嵌进去。

当前决定不把 full terminal layout / full hub layout 拉成主线变量。
如果后续要做，也只作为扩展里的 template selection，而不是第一篇的主模型。

### 6. 已冻结的实验路线

实验按六步走，顺序不能乱。

#### EXP-0：factor screening

目的：验证 factor families 对 A/S/F 的主导关系。

#### EXP-6：ASF 对抗性证伪（可选）
