# 模型建模全解：参数→现实含义→算法→实验

## 一、现实场景是什么

想象一座城市（比如深圳），有 8 个垂直起降场（vertiport），分布在城区各处。我们要设计一张**固定走廊网络**——决定哪些起降场之间开通空中走廊、走廊上跑多少航班。

每个起降场不是一个简单的点。它有自己的**运行构型**：
- 几个起降坪（pad）在同时服务
- 飞机到了得在空中等待环（holding ring）排队
- 从不同方向来的走廊接的是不同的进近口（interface）
- 等待环占用的空域可能挡住旁边经过的别的走廊
- 运行时有噪声、影响周围居民

**核心矛盾**：上层网络设计需要知道终端的特性来做决策，但终端太复杂不能整个塞进优化器。所以要"抽象"——问题是抽象到什么程度够用。

## 二、模型中每个实体对应什么

### 2.1 终端样式（TerminalStyleConfig）——一个起降场怎么运行

| 代码参数 | 现实含义 | 典型值 |
|----------|----------|--------|
| `padCount` | 起降坪数量。类似机场跑道数，同时能服务几架飞机 | 2-4 个 |
| `waitingSlots` | 空中等待位数量。排队满了新来的飞机就得转圈或 go-around | padCount×2 |
| `serviceTime` | 单次起降服务时间（秒）。飞机落地→卸客→装客→起飞 | 90-150 秒 |
| `ringRadiusNm` | 等待环半径（海里）。飞机在空中绕圈等待的圆的半径 | 0.3-1.5 nm |
| `approachLengthNm` | 进近航段长度（海里）。从走廊末端到起降坪的路程 | 0.5-1.5 nm |
| `feasibleCorridors` | 这个终端构型允许连接哪些走廊。不是所有方向都能接 | 看具体布局 |
| `footprintBlock` | 这个终端的空域脚印会挡住哪些走廊。等待环+进近航段占了空域 | 看 ringRadius |
| `noiseIndex` | 噪声水平（无量纲）。运行时对周围的噪声影响 | 0.5-1.5 |
| `populationExposure` | 影响人口数（人）。噪声覆盖范围内的居民数 | 0-50 |
| `interfaceDirections` | 每个进近口的方向标签："direct"=直线进近，"detour"=绕行进近 | 约 70% direct, 30% detour |
| `marginalExtCoeff` | 每接口每单位流量的边际外部性系数 χ_h。流量越大噪声越大 | 0-0.5 |
| `baseExternality` | 终端固有外部性 X̄_t（不管有没有航班都存在的影响） | 0-2 |

### 2.2 走廊（CandidateCorridor）——两个起降场之间的一条空中通道

| 代码参数 | 现实含义 | 城市级实验中怎么生成 |
|----------|----------|---------------------|
| `id` | 走廊编号（E1, E2, ...） | 自动编号 |
| `origin`, `destination` | 走廊连接的两个终端 | 距离<15nm的终端对 |
| `cost` | 单位流量运营成本。正比于走廊距离 + 随机扰动 | distance × (0.8+0.4×rand) |
| `activationCost` | 走廊启用固定成本。开通走廊的一次性投入 | distance × 0.5 |
| `capacity` | 走廊容量上界（架次/小时） | 20-50 |
| `region` | 空间方位标签（east/north/west/south），用于脚印阻塞判断 | 由终端间角度决定 |

### 2.3 需求场景（DemandScenario）

| 代码参数 | 现实含义 |
|----------|----------|
| `odDemand` | OD 对需求量（架次/小时）。T1→T3 要 5 架/小时这种 |
| `probability` | 场景发生概率（当前单场景=1.0） |

城市级实验中：40% 的 OD 对有需求，每对需求 = baseDemand × (0.5+rand)。

### 2.4 网络实例（NetworkInstance）

把上述三者打包：终端列表 + 走廊列表 + 需求场景 + 样式选项 + 未满足需求惩罚 κ=100。

## 三、终端中观模型（MesoscopicModel）——从构型参数算出终端特性

### 3.1 容量 μ

```
μ = padCount × 3600 / serviceTime    (架次/小时)
```

例：3 个起降坪、120 秒服务时间 → μ = 3×30 = 90 架次/小时。这是这个终端同时能处理的最大航班率。

### 3.2 延迟函数 D(λ)

现实中：飞机到达率 λ 越接近容量 μ，排队越长、等待越久。模型用参数化形式：

```
D(λ) = α × (λ / (μ - λ))^β
```

这个形式来自排队论（M/M/c 类型），λ/(μ-λ) 是"利用率比"。α 和 β 通过精确计算拟合得到。

**精确计算考虑三个成分**：
1. **起降坪排队延迟**（M/D/c 近似）：多服务台、确定性服务时间的排队延迟
2. **环形等待延迟**：与 ringRadius 正比，负载率高时飞机在空中多绕几圈
3. **合流冲突延迟**：等待位快满时的非线性延迟放大

### 3.3 接口级延迟 D_{t,h}(λ_h)

每个终端有多个进近接口（每条走廊对应一个接口）。每个接口独立拟合延迟参数。

**关键差异来源**：`interfaceDirections`
- **"direct"接口**：直线进近，延迟正常
- **"detour"接口**：绕行进近，**延迟放大 3 倍**（ringRadius 加倍 + delayScale=3）

这意味着同一个终端，从不同方向来的飞机等待时间可以差很多。A0 看不见这个差异（它只看聚合延迟），A1 能看见。

### 3.4 接口容量上限 capMax_h

```
capMax_h = μ / nH    (终端总容量 / 接口数)
```

每个接口平分容量。这是每个接口的"不能超过"的流量上限。

### 3.5 聚合延迟 D_agg（A0 专用）

A0 不区分接口，只看总流量。D_agg 通过"假设流量均匀分配到各接口"来拟合标量延迟函数。

**信息损失点**：如果实际流量不是均匀分配的（比如全集中在 detour 接口），D_agg 会低估或高估延迟。

### 3.6 空域脚印

```
footprintRadius = ringRadius + approachLength    (海里)
blockedCorridors = footprintBlock                (直接从配置读取)
```

等待环 + 进近航段占用的空间。如果一条走廊恰好从这片空域穿过，就会被阻塞。A0/A1 看不见这个，A2/Full 能看见。

### 3.7 外部性

```
X_t = X̄_t + Σ_h χ_h × λ_h
```

基础外部性（终端存在本身的影响）+ 各接口流量带来的边际外部性。流量越大，噪声影响越大。

### 3.8 无量纲化参考值

```
L^ref = (0.5μ) × D_agg(0.5μ)    — 50% 负荷率下的延迟量作为基准
X^ref = X̄_t + Σ(χ_h) × 0.5μ/nH — 50% 负荷率下的外部性作为基准
```

用来把不同终端的 Ψ_t 归一化到可比较的量纲。

## 四、终端广义成本 Ψ_t

```
Ψ_t = η × L̃_t + ξ × X̃_t
```

| 符号 | 含义 | 当前值 |
|------|------|--------|
| η | 延迟权重 | 1.0 |
| ξ | 外部性权重 | 0.1 |
| L̃_t = L_t / L^ref | 无量纲化总延迟负荷 | — |
| X̃_t = X_t / X^ref | 无量纲化外部性 | — |

**L_t 的接口分解版本**：
```
L_t = Σ_h λ_h × D_{t,h}(λ_h)
```
每个接口的流量 × 该接口的延迟。是非线性凸函数（流量越大延迟指数增长）。

**L_t 的聚合版本（A0 用）**：
```
L_t = λ_total × D_agg(λ_total)
```
总流量 × 聚合延迟。不区分接口。

## 五、优化问题——MIP 在做什么决策

### 5.1 决策变量（要决定什么）

| 变量 | 类型 | 含义 |
|------|------|------|
| x(e) ∈ {0,1} | 二值 | 走廊 e 是否开通 |
| f(e,ω) ≥ 0 | 连续 | 走廊 e 在场景 ω 下跑多少流量 |
| ψ(t,ω) ≥ 0 | 连续 | 终端 t 在场景 ω 下的广义成本 |
| unmet(ω) ≥ 0 | 连续 | 场景 ω 下有多少需求没满足 |
| z(t,k) ∈ {0,1} | 二值 | 终端 t 选第 k 种样式（当前每终端1种，退化） |

### 5.2 目标函数（最小化总成本）

```
min J = Σ_e activationCost_e × x_e                        ← 开通走廊的固定投入
      + Σ_ω p_ω × [ Σ_e cost_e × f_e^ω                   ← 走廊运营的流量成本
                   + Σ_t ψ_t^ω                              ← 终端区的延迟+外部性成本
                   + 100 × unmet^ω ]                        ← 没满足的需求的惩罚
```

**现实含义**：权衡"开多少走廊"（固定成本高）、"每条走廊跑多少"（运营成本）、"终端承受多大负荷"（拥挤延迟）、"能不能满足需求"（社会损失）。

### 5.3 约束（什么是不允许的）

| 约束 | 公式 | 现实含义 |
|------|------|----------|
| C1 | f_e ≤ cap_e × x_e | 没开通的走廊不能跑流量；开通的也有容量上限 |
| C2 | x_e ≤ Σ feasCoeff × z | 走廊只能在两端终端的样式都允许时才能开通 |
| C3 | Σ_k z(t,k) = 1 | 每个终端选且只选一种运行样式 |
| C4 | Σ_e f_e + unmet ≥ demand | 走廊流量 + 允许的未满足量 ≥ 总需求 |
| C5 | λ_{t,h} = Σ_{e∈h} f_e | 终端接口 h 的负荷 = 从该接口进出的所有走廊流量之和 |
| C6a | Σ_h λ_{t,h} ≤ μ_t | 终端总负荷不超过总容量 |
| C6b | λ_{t,h} ≤ capMax_h | 每个接口负荷不超过该接口容量 |
| C7 | ψ_t = PwL(λ) | 终端成本 = 延迟函数的分段线性近似 |

**C2 是抽象层级的关键差异点**：
- A0: feasCoeff 恒为 1 → 所有走廊都可以开 → 优化器看不到接口限制
- A1/Full: feasCoeff 根据接口矩阵和脚印阻塞来定 → 有些走廊被禁止

**C7 是第二个差异点**：
- A0: 总负荷的一元分段线性化 → 看不到接口级延迟差异
- A1/Full: 每接口独立分段线性化 → 能区分 direct 和 detour 口的延迟差

## 六、分段线性化（PwL）——怎么把非线性函数塞进 MIP

L(λ) = λ × D(λ) 是非线性凸函数，MIP 只能处理线性。做法：

1. 在 [0, 0.95×capMax] 上取 8 个均匀断点
2. 算出每个断点处的精确 L 值
3. 相邻断点之间用直线连接（弦插值）
4. 用增量变量 δ_j 和二值排序变量 y_j 编码"当前在哪一段"

**问题（Codex 诊断的根因）**：弦插值对凸函数是**上界**。MIP 看到的终端成本比真实的高。Full 按接口分解后每段曲线更陡 → 高估更严重 → 导致负 regret。

---

## 七、评估器（Evaluator）——怎么判断方案好不好

拿到 MIP 求出的方案后，不用 PwL 近似，用精确的 D(λ) 重新算目标：

1. 走廊激活成本：直接累加
2. 流量成本：检查走廊两端是否真的可行（Full 视角）——不可行的流量变成 unmet 惩罚
3. 终端成本：用接口级精确 L_t = Σ_h λ_h × D_h(λ_h) 算，不用 PwL
4. 外部性：ξ>0 时通过 Ψ_t 算入

**Regret = Evaluator(A0方案, Full视角) - Evaluator(Full方案, Full视角)**

## 八、城市级实验具体怎么做的

### 8.1 实例生成（InstanceLibrary.buildCityInstance）

1. **放终端**：8 个点随机撒在 20×20 nm² 区域
2. **连走廊**：距离 < 15 nm 的终端对生成走廊（约 20-27 条）
3. **分配样式**：每终端 1 种固定样式
   - padCount: 2-4（随机）
   - serviceTime: 90-150 秒（随机）
   - ringRadius: 0.3（无脚印）/ 1.0（moderate）/ 1.5（severe）
   - interfaceDirections: 70% direct, 30% detour（随机）
   - marginalExtCoeff: 0-0.5（随机）
   - footprintBlock: none时无 / moderate时30%终端阻塞1条 / severe时60%终端阻塞1-2条
4. **生成需求**：40% OD 对有需求，baseDemand × (0.5+rand)
5. **构建 NetworkInstance**

### 8.2 求解流程（RegretFramework.computeRegret）

对每组参数(demand, footprint, seed)：

1. **预计算**：MesoscopicModel 为每个终端算出完整 TerminalResponse
2. **Full 求解**：Full plugin → TwoStageMIP → y*（含 Ψ_t 的 PwL 近似）
3. **Full 评估**：Evaluator(y*, Full) → J*（精确计算）
4. **A0 求解**：A0 plugin → TwoStageMIP → ŷ_A0（更松的约束 + 标量 PwL）
5. **A0 评估**：Evaluator(ŷ_A0, Full) → J(ŷ_A0)（用 Full 精确评估 A0 的方案）
6. **Regret**：Δ_A0 = J(ŷ_A0) - J*
7. 同理计算 A1、A2 的 regret

### 8.3 参数扫描

| 维度 | 取值 |
|------|------|
| demand | low(base=3), medium(base=8), high(base=15) |
| footprint | none(0%阻塞), moderate(30%阻塞1条), severe(60%阻塞1-2条) |
| seed | 1,2,3,4,5 |

共 3×3×5 = 45 轮。每组(demand, footprint)取 5 种子的平均 regret，填入 3×3 的 regime map。
