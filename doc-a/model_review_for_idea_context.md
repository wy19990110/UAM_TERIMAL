## 一、我对问题的理解（请审阅是否正确）

### 1.1 核心问题

给定一组候选外部走廊和多个终端区，设计最优航路网络（选哪些走廊、每个终端用什么管理样式、流量怎么分配）。终端区通过"抽象接口"进入上层网络设计，抽象层级不同（A0/A1/A2/A2+），上层设计的最优解可能不同。用 regret Δ_ℓ = J(ŷ_ℓ; M) - J(y\*; M) 衡量抽象层级是否"够用"。

### 1.2 我理解的决策变量

- x_e ∈ {0,1}：走廊 e 是否激活
- z_{tk} ∈ {0,1}：终端 t 是否选样式 k（每终端恰好选一个）
- f_e^ω ≥ 0：场景 ω 下走廊 e 的流量
- unmet^ω ≥ 0：场景 ω 下未满足需求

### 1.3 我理解的目标函数

1.md 定义（第 791-805 行）：

```
J(x,z,f;φ) = F^route(x) + F^term(z) + Σ_ω p_ω Q^ω(x,z,f;φ)

Q^ω = Σ_e c_e f_e^ω + Σ_t Ψ_t(λ_t^ω; φ_t) + P^unmet(ω)
```

其中 Ψ_t(λ_t; φ_t) 是**终端区广义成本**，依赖到达率 λ_t 和抽象接口 φ_t。

## 二、我实际实现了什么

### 2.1 实际目标函数

```
J_impl = Σ_e activationCost(e)·x(e)
       + Σ_ω p_ω · [Σ_e cost(e)·f(e,ω) + penalty·unmet(ω)]
```

**对照 1.md：**
- F^route(x) = Σ activationCost·x → ✅ 已实现
- F^term(z) → ❌ **未实现**（没有终端样式选择成本）
- Σ c_e f_e^ω → ✅ 已实现
- P^unmet(ω) → ✅ 已实现
- **Ψ_t(λ_t; φ_t) → ❌ 完全缺失**

### 2.2 实际约束

| 约束 | 数学形式 | 状态 |
|------|----------|------|
| C1: 走廊容量 | f(e,ω) ≤ cap(e)·x(e) | ✅ 已实现 |
| C2: 走廊可行性 | x(e) ≤ Σ_k feasCoeff(t,k,e)·z(t,k) | ✅ 已实现（通过 plugin） |
| C3: 样式唯一 | Σ_k z(t,k) = 1 | ✅ 已实现 |
| C4: 需求满足 | Σ_e f(e,ω) + unmet(ω) ≥ demand(ω) | ✅ 已实现 |
| **C5: 终端容量** | **Σ_{e∈inbound(t)} f(e,ω) ≤ μ_t** | ❌ **缺失** |
| **C6: 延误成本** | **Ψ_t(λ_t; φ_t) 进入目标** | ❌ **缺失** |

### 2.3 缺失的 Ψ_t 为什么致命

Ψ_t 是终端区广义成本，应至少包含延误成本 D_t(λ_t)。它是 φ_t 的函数——不同抽象层级看到的 Ψ_t 不同：
- A0 下：Ψ_t 只能用 (μ,D) 计算
- A1 下：Ψ_t 可以额外考虑接口约束对流量分布的影响
- A2 下：Ψ_t 还可以包含外部性成本

**没有 Ψ_t，A0 的"容量抽象"名不副实**——μ 和 D 都被计算了但从未影响任何决策。当前 E0/E1 的 regret 全部来自"接口/阻塞的硬切"，不来自"延误/容量的渐变差异"。

### 2.4 Ψ_t 难在哪里

Ψ_t(λ_t; φ_t) 是**流量 f 的非线性函数**（因为 λ_t = Σ_{e→t} f_e，而 D(λ) = α·(λ/(μ-λ))^β 是非线性的）。直接放入 intlinprog（线性 MIP）需要线性化或分段近似。

## 三、终端区中观模型：我实现了什么

### 3.1 容量计算

```
μ = min(padCount × 3600/serviceTime, waitingSlots × 3600/serviceTime)
```

取起降坪吞吐和等待位吞吐的瓶颈。

**问题待确认**：这个公式是否合理？还是应该用更复杂的排队论容量（如 Erlang 损失模型）？

### 3.2 延误计算

三部分之和，拟合为参数化函数 D(λ) = α·(λ/(μ-λ))^β：

1. **起降坪排队延误**：M/D/c 近似（Cosmetatos 近似：W_q(M/D/c) ≈ W_q(M/M/c)/2）
2. **环形等待延误**：ringCircuit × ρ/(1-ρ+0.1)，ringCircuit = 2π·ringRadius·60/60
3. **合流延误**：10 × min(1, λ·serviceTime/3600/waitingSlots) 秒

**问题待确认**：
- 环形延误公式的物理含义是否正确？
- 合流延误的 10 秒上限是否合理？
- M/D/c 近似对终端区操作是否合适（服务时间真的是确定性的吗）？

### 3.3 空域脚印

```
radius = ringRadiusNm + approachLengthNm
blockedCorridors = styleConfig.footprintBlock  % 手工指定，非几何计算
```

**问题**：当前阻塞关系是**手工硬编码**在 StyleCatalog 里的。城市级实验需要基于坐标的几何阻塞判断。

### 3.4 外部性

```
externalityCost = w1 × noiseIndex + w2 × populationExposure
```

默认权重 [1, 1]。

**问题**：外部性只在 Evaluator 里算了（事后评估），没有进入 MIP 目标函数。

## 四、抽象层级插件：信息可见性

| 方法 | A0 | A1 | A2 | A2+ | Full |
|------|:--:|:--:|:--:|:---:|:----:|
| getCapacity(t,k) → μ | ✓ | ✓ | ✓ | ✓ | ✓ |
| getDelay(t,k,λ) → D(λ) | ✓ | ✓ | ✓ | ✓ | ✓ |
| isCorridorFeasible(t,k,e) | **恒true** | 查A | 查A | 查A | 查A |
| isCorridorBlocked(t,k,e) | **恒false** | **恒false** | 查V | 查V | 查V |
| getExternalityCost(t,k) | **恒0** | **恒0** | 查X | 查X | 查X |
| isVehicleQualified(t,k,v) | **恒true** | **恒true** | **恒true** | 查C | 查C |

**关键矛盾**：A0 声称只看 (μ,D)，getCapacity 和 getDelay 确实对所有层级都返回真值。但由于 MIP 和 Evaluator **根本不调用 getCapacity/getDelay**，A0 的 (μ,D) 实际上是**死代码**。当前 A0 的行为等同于"什么都看不见"，而非"只看容量和延误"。

## 五、E0/E1 实验设计

### 5.1 E0 实例

- T1 固定 fixed_north（允许 N,B），T2 固定 fixed_south（允许 S,B）
- 走廊 N(cost=1), S(cost=4), B(cost=2.5)，容量各 10
- 需求 5.0 on T1→T2
- 每个终端只有一种样式（z 形同虚设）

**Regret 来源**：纯接口可行性的硬切（N 在 T2 不可行，A0 看不见）

### 5.2 E1 实例

- T1 固定 compact_ring（不阻塞），T2 固定 wide_ring（阻塞 C）
- 走廊 C(cost=1), N(cost=2), S(cost=2)，容量各 10
- 需求 5.0 on T1→T2
- 每个终端只有一种样式

**Regret 来源**：纯空域脚印阻塞的硬切（C 在 T2 被阻塞，A1 看不见）

### 5.3 实验局限

1. **没有"渐变"regret**——当前 regret 全是"全对 or 巨额惩罚"，没有中间地带
2. **没有容量/延误驱动的 regret**——因为 Ψ_t 缺失
3. **终端样式不是决策变量**——z 始终被固定
4. **规模太小**——2 终端 3 走廊
5. **单 OD 对**——没有多商品流
