# UAM_NET 项目阶段性评估报告

**评估日期**: 2026-03-31  
**评估范围**: Phase 1–3 全部代码、E0/E1 实验结果、43 个自动化测试  
**评估目标**: 判断当前进展能支撑什么、不能支撑什么，以及下一步优先级

---

## 一、当前实验结果能支撑什么结论

### 1.1 已稳固确立的结论

**结论 1: 容量抽象 (A0) 不总是充分的。**

E0 用了一个精心设计的极小反例：T1 只允许 {N, B}，T2 只允许 {S, B}，两种终端样式的 (μ, D) 完全相同。A0 因为看不见接口限制，选择了成本最低的走廊 N，但 N 在 T2 端不可行，导致真值评估时流量全部变为未满足需求惩罚，Δ_A0 = 487.5。A1 能看见接口约束，正确选择走廊 B，Δ_A1 = 0。

这是一个干净的存在性证明。P2（A0 不充分反例）已被实验验证。

**结论 2: 接口抽象 (A1) 也不总是充分的。**

E1 构造了 compact_ring 与 wide_ring 两种样式，(A, μ, D) 完全相同，仅 V（空域脚印）不同。wide_ring 阻塞走廊 C。A0 和 A1 都看不见阻塞，选择了最便宜的走廊 C，Δ = 490。A2 能看见阻塞，正确避开 C，Δ_A2 = 0。

P3（A1 不充分反例）也已被实验验证。

**结论 3: 抽象层级确实形成严格嵌套的信息递进。**

代码中 A0 ⊂ A1 ⊂ A2 ⊂ Full 的插件结构清晰。每一层只比上一层多暴露一类信息（接口 A → 空域脚印 V/外部性 X → 资格/关闭 C）。实验结果与理论预测完全吻合。

### 1.2 这些结论的强度评估

E0 和 E1 都是**构造性反例**（constructive counterexamples），不是统计性实验。它们的逻辑是：

- 存在一个实例，使得 A0 失败而 A1 成功
- 存在一个实例，使得 A1 失败而 A2 成功

这种证据的长处是**干净、无噪声、完全可复现**。短处是：只能证明"不总是充分"，无法回答"多经常不充分"、"在什么参数区间内不充分"。

## 二、当前实验结果不能支撑什么

### 2.1 无法支撑的结论

1. **"A1 通常就够了"或"A2 通常才够"** —— 没有参数扫描，不知道参数区间。
2. **"抽象层级的选择对现实城市网络有显著影响"** —— 只有 2 终端 3 走廊的玩具实例，没有城市级验证。
3. **"regret 与输出误差之间存在可预测的定量关系"** —— P4（Δ ≤ 2ε_J）在代码中还没有实现为可验证的数值实验。
4. **"终端样式选择是决策变量时，抽象层级仍然重要"** —— E0/E1 中终端样式都是外生固定的，不是求解器的决策变量。虽然 TwoStageMIP 中有 z 变量支持样式选择，但实际实验中每个终端只有一种样式。
5. **"分阶段设计在正确接口下可以恢复"** —— 这是论文的核心工程主张，但目前没有任何实验直接对比"联合设计 vs 带正确接口的分阶段设计"。
6. **"A2+ 有必要"** —— 没有做异构终端区实验。

### 2.2 关键缺口排序

| 缺口 | 严重程度 | 论文中必须有？ |
|------|---------|--------------|
| 无参数区间/regime map | 致命 | 是（WP6 核心交付物） |
| 终端样式作为决策变量的实验 | 高 | 是（否则 z 变量形同虚设） |
| 城市级规模验证 | 高 | 是（否则 reviewer 会质疑 scalability） |
| P4 数值验证 | 中 | 强烈建议（理论主线的闭环） |
| A2+ 异构终端实验 | 低 | 可选加分项 |

## 三、代码框架评估

### 3.1 架构优点

1. **包结构清晰**：`+uam/+core`、`+abstraction`、`+solver`、`+terminal`、`+experiments`、`+regret` 各司其职。
2. **插件模式设计正确**：`TerminalPlugin` 抽象基类 + A0/A1/A2/Full 具体插件，新增抽象层级只需新建一个 Plugin 类。
3. **Regret 框架是通用编排器**：`RegretFramework.computeRegret()` 不硬编码任何具体实例，可以接任意 `NetworkInstance`。
4. **MIP 建模使用 `optimproblem` + `intlinprog`**：比手写矩阵可读性好得多，后续扩展约束方便。
5. **测试覆盖度好**：43 个测试涵盖核心数据结构、抽象层级、求解器、终端模型、端到端实验。

### 3.2 架构问题

**问题 1: MIP 求解器的扩展性瓶颈。**

`TwoStageMIP.buildProblem()` 中约束 C2（走廊可行性）使用三重嵌套循环：对每个终端、每条走廊、每种样式。当前 2×3×1 = 6 次迭代，但城市级（12 终端 × 150 走廊 × 3 样式 = 5400 次 + 对应约束变量）将产生大量约束。`intlinprog` 对大规模整数规划可能力不从心。

**建议**: 预计城市级需要切换到 Gurobi 或 CPLEX。当前 `optimproblem` 接口支持 `solvers` 选项，迁移成本可控。

**问题 2: 延误模型在目标函数中被忽略。**

`Evaluator.evaluate()` 和 `TwoStageMIP` 的目标函数中，延误 D(λ) 没有出现。当前目标函数是：

```
J = Σ activationCost × x + Σ_ω p_ω × [Σ cost × f + penalty × unmet]
```

延误只在 `TerminalResponse.computeDelay()` 中计算，但没有被求解器调用。这意味着 A0 的 (μ, D) 中 D 实际上不影响决策。容量 μ 也没有直接进入约束（走廊有自己的 capacity 上限，但终端容量没有作为约束出现）。

**后果**: 当前的 regret 差异完全来自接口可行性和阻塞，不来自容量和延误差异。这削弱了 A0 → A1 差异的解释丰富性——A0 不是因为"延误估计不准"而失败，而是因为"完全看不见接口"而失败。

**建议**: 在目标函数中加入终端延误项，在约束中加入终端容量上限。否则论文中关于"容量抽象"的讨论实际上名不副实——A0 的容量信息根本没有被使用。

**问题 3: 空域脚印的阻塞判断是硬编码的。**

`FootprintCalc.compute()` 直接返回 `styleConfig.footprintBlock`，没有做任何几何计算。虽然计算了 radius，但 radius 没有被用来判断哪些走廊被阻塞。阻塞关系完全由人工在 `StyleCatalog` 中指定。

**后果**: 对于预设的 E0/E1 反例没有问题。但城市级实验中，走廊与终端空域脚印的空间关系应该通过几何判断自动确定，而不是手工枚举。

**建议**: 实现基于坐标的几何阻塞判断。为走廊和终端添加空间位置属性。

**问题 4: 需求模型过于简化。**

当前 OD 需求是单一数值，不分方向、不分时段。走廊的 origin/destination 固定。没有多商品流（multi-commodity flow）结构——所有需求走所有走廊。

**后果**: 城市级实验需要多 OD 对、多走廊路由，当前的单商品流模型不够。

**问题 5: 无可视化。**

`results/figures/` 目录为空。没有任何绘图脚本。论文需要：regret 柱状图/热力图、网络拓扑图、regime map。

### 3.3 框架能否支撑城市级实验？

**短答案: 架构可以，但需要补充以下功能模块。**

| 模块 | 当前状态 | 城市级所需 |
|------|---------|-----------|
| 多 OD 对 + 多商品流 | 缺失 | 必须 |
| 终端容量约束 | 缺失 | 必须 |
| 延误进入目标函数 | 缺失 | 必须 |
| 几何阻塞判断 | 缺失 | 必须 |
| 终端样式作为真正的决策变量 | 有框架但未测试 | 必须 |
| Gurobi/CPLEX 接口 | 缺失 | 强烈建议 |
| 可视化脚本 | 缺失 | 必须 |
| 实例生成器（参数化城市级网络） | 缺失 | 必须 |
| 参数扫描 + regime map | 缺失 | 必须 |

## 四、下一步优先级建议

### 第一优先级（1-2 周）：修复目标函数

将延误和终端容量正确纳入 MIP 模型。否则当前的 A0 定义（容量抽象）名不副实。这是理论一致性问题，不能拖。

修复后重新运行 E0/E1，预期结果不变（因为 E0/E1 的 regret 来源是接口/阻塞，不是延误），但框架变得完整。

### 第二优先级（2-3 周）：终端样式作为决策变量的实验 (E2)

设计一个实例：同一终端可选 compact_ring 或 wide_ring，让求解器同时优化走廊选择和终端样式选择。这是 TwoStageMIP 中 z 变量存在的意义，必须有一个实验使用它。

### 第三优先级（3-5 周）：城市级实例生成器 + 中等规模实验 (E3)

实现参数化的城市级网络生成器（8-12 终端、80-150 走廊、多 OD 对）。需要多商品流模型、几何坐标、自动阻塞判断。然后做参数扫描，生成 regime map。

### 第四优先级（并行）：可视化

至少需要：
- E0/E1/E2 的 regret 柱状图
- 网络拓扑对比图（不同抽象层级选择的不同网络）
- 城市级 regime map（热力图或参数区间图）

### 第五优先级（并行）：P4 数值验证

在 E0/E1 基础上，数值计算 ε_J（输出误差上界），验证 Δ ≤ 2ε_J 是否成立。这是理论主线的闭环。

## 五、阶段性实验报告（简明版）

### 项目名称
面向低空可复用航路网络设计的终端区决策保持抽象

### 已完成里程碑

| 阶段 | 内容 | 状态 |

| G2: 抽象层级必要性 | E1 实验：A1 失败、A2 成功 | ✅ 完成 |
| 代码基础设施 | 43 个自动化测试全部通过 | ✅ 完成 |

### E0 实验结果

**场景**: 2 终端 3 走廊。T1=fixed_north（允许 N, B），T2=fixed_south（允许 S, B）。1 个需求场景，需求量 5.0。

| 层级 | 选择的走廊 | 抽象下目标值 | 真值评估 | Regret (Δ) | 拓扑一致 |
|------|-----------|------------|---------|-----------|---------|
| Full (真值) | B | — | 14.5* | 0 | — |
| A0 | N | ~5.0 | 502.0 | 487.5 | 否 |
| A1 | B | 14.5 | 14.5 | 0 | 是 |

*注: J* = activationCost(0) + cost_B(2.5) × flow(5.0) + unmet(0) + externality(2.0) = 14.5

**机制解释**: A0 看不见接口约束，以为所有走廊对所有终端都可行，选了成本最低的 N。但 N 在 T2 端不可行（T2 只允许 S 和 B），导致流量在真值评估中变为未满足需求惩罚（100 × 5.0 = 500）。

### E1 实验结果

**场景**: 2 终端 3 走廊。T1=compact_ring（不阻塞），T2=wide_ring（阻塞 C）。1 个需求场景，需求量 5.0。

| 层级 | 选择的走廊 | 抽象下目标值 | 真值评估 | Regret (Δ) | 拓扑一致 |
|------|-----------|------------|---------|-----------|---------|
| Full (真值) | N, S | — | 62.5* | 0 | — |
| A0 | C | ~7.0 | 552.5 | 490.0 | 否 |
| A1 | C | ~7.0 | 552.5 | 490.0 | 否 |
| A2 | N, S | 62.5 | 62.5 | 0 | 是 |

*注: J* 包含外部性成本（wide_ring 的 noiseIndex=1.5 和 populationExposure=50）

**机制解释**: compact_ring 和 wide_ring 有相同的接口集合 {C, N, S}，但 wide_ring 的大环形空域脚印阻塞了走廊 C。A0 和 A1 都看不见阻塞信息，选了最便宜的 C。A2 能看见阻塞，正确避开 C 改选 N+S。

### 当前局限

1. **仅有构造性反例，无参数区间分析**。不知道"多经常"会出现层级不足的情况。
2. **延误和终端容量未进入优化目标**。A0 的"容量抽象"名不副实——容量信息实际上没有影响决策。
3. **终端样式均为外生固定**。没有测试终端样式作为决策变量的场景。
4. **仅 2 终端 3 走廊**。规模太小，无法说明对现实网络设计的影响。
5. **无可视化**。

### 下一阶段工作计划

1. 修复目标函数（纳入延误和终端容量约束）
2. 实现终端样式选择实验 (E2)
3. 城市级实例生成器 + 中等规模实验 (E3)
4. 可视化和 regime map
5. P4 数值验证
