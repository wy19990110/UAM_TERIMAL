# 一、先用一句话定义这篇论文

**这篇论文研究的是：在低空可复用航路网络设计里，异质终端不能再被当成普通点节点；问题不是把终端整体抬升进一个更大的联合优化器，而是识别一个对网络设计“足够有用”的终端接口层级。**
当前采用的接口假设是 A/S/F：A 是可接性（access admissibility，可接性），S 是端口级服务能力（service capability，服务能力），F 是终端邻域的脚印/冲突（footprint/conflict，脚印/冲突）。整篇论文真正交付的是**接口充分性（interface sufficiency，接口充分性）与模型升级规则（model-upgrade rules，模型升级规则）**，不是终端区运行设计本身。

# 二、问题背景、研究动机、为什么值得做

## 2.1 背景

低空交通 / UAM（城市空中交通）正在从点状示范走向网络化建设。这个阶段真正要回答的是：城市里哪些 reusable routes / corridors（可复用航路/走廊）应该建、怎么接 terminal、以及流量如何分配。TRC 关心的正是这种**新兴技术对交通系统规划、设计、运营、控制的影响**，而不是技术本身。官方 scope 明确强调，重点是 emerging technologies 对 transportation system planning / design / operation / control 的含义，而不是技术本体。([科学直接][1])

## 2.2 核心矛盾

外部航路网络设计希望 terminal 越粗越好，最好就是“一个点 + 一个总容量”；但现实 terminal 明显异质：
有的 terminal 更 procedure-like（程序型），有的更 path-based（路径型）；
port 朝向、接入规则、局部服务能力、近端保护区、等待环/合流结构都不一样。
所以问题不是“终端重要吗”，而是：

> **为了把网络设计做对，终端最少要向上层暴露什么信息？**

这正是交接总述里已经冻结的主线：不是永久维持联合优化，而是先联合验证，再恢复工程分层设计。

## 2.3 文献断层

当前文献大致分成四块，但它们之间正好有一个空档：

1. **战略网络设计 / 枢纽网络设计（strategic network design / hub network design）**
   这类研究很成熟，常常考虑拥堵、容量、服务水平，但通常把 terminal/hub 的内部异质性压成成本、容量或服务水平，而不直接处理“接入规则 + port 级差异 + 局部脚印”这种接口问题。Elhedhli & Hu 2005、Khosravi 2024、Bayram 等的工作说明：在战略网络设计里，用 congestion/capacity/service-level surrogates（拥堵/容量/服务等级代理）是完全正当的。([科学直接][2])

2. **空管 / 机场拥堵与排队建模（ATM/airport congestion and queueing）**
   这条线说明：机场/终端区内部的等待、资源竞争、拥堵，可以用 analytical queueing theory（解析排队论）、stochastic modeling（随机建模）和拥堵近似来建模。Shone 等的综述把 analytical queueing theory、stochastic optimal control、robust optimization 等列为空管/机场拥堵的主工具；Peterson 1995、Itoh 2019 等工作也都把机场/进近拥堵明确建成 queueing-based models（基于排队的模型）。([科学直接][3])

3. **UAM / 低空空域设计（UAM/low-altitude airspace design）**
   这条线说明：城市低空空域的 geometry、capacity、procedures、legacy constraints（几何、容量、程序、既有空域约束）是网络设计的一等公民。Bauranov 2021 的综述强调 airspace geometry/capacity 是核心结构因素；Vascik 2019 的 geometric airspace assessment 直接把受保护空域、机场程序、瓶颈和可用空域放进分析框架。([科学直接][4])

4. **geofencing / terminal-area control / vertiport operations（围栏、终端区控制、起降场运行）**
   Kim 2022 直接把 geofence volumes（围栏体积）作为低空路径规划中的 keep-in/keep-out 约束；Shao 2021、Song 2022 这类 terminal-area control 研究则明确讨论 multi-ring structure、holding circles、junction control rules、approach control 等 terminal 邻域控制结构。也就是说，局部空域占用、近端保护区、等待环这类对象不是你们凭空发明的。([MDPI][5])

**这篇论文要补的空档就是：如何把 terminal 的内部异质性压缩成一个“网络设计可直接调用”的中层接口。**

# 三、这篇论文的核心立场

## 3.1 不是发现 ASF，而是采用 ASF 作为先验接口假设

A/S/F 不是从数据里“聚类发现”的，也不是让 EXP-0 去“发现”的。
它是一个**需求牵引 + 文献脉络 + 工程分层需要**共同驱动的先验接口假设：

* A：终端从哪些方向、哪些 connector、哪些 port 可以接入；
* S：terminal 内部服务能力与拥堵如何通过 port 级边际代价表现出来；
* F：terminal 周边哪些边会因为保护区、近端控制或局部冲突而被挡/被罚。

所以 EXP-0 的角色不是“发现 ASF”，而是对 adopted interface hypothesis（采用的接口假设）做 screening / sanity check（筛查 / 合理性检查）。这是你们后续写作里必须说清楚的第一条边界。

## 3.2 不是证明全局最小性，而是检验在 adopted truth class 中的 tested sufficiency

这篇论文**不该**写成“我们证明了 ASF 是全局最小充分接口（global minimum sufficient interface）”。
实验撑不住这句话。
更稳的说法是：

> **在本文采用的终端真值类（adopted terminal truth class）、候选图族（candidate graph families）和网络设计问题类（decision class）下，A/S/F 形成了一个可检验的分层接口；我们关心的是 tested interface sufficiency（经检验的接口充分性）和 model-upgrade rules（模型升级规则）。**

这意味着整篇论文的标题、摘要、结论都应该围绕：

* terminal 不是点；
* A/S/F 这三条通道会独立误导网络设计；
* 在什么条件下应该从 M0 升到 M1，再升到 M2。

而不是围绕“最小性定理”。

# 四、方法主线：我们到底在建什么模型

这部分是新上下文最需要掌握的。

## 4.1 最上层：网络设计对象是什么

网络设计对象始终是：

* 选 backbone / corridor（主干边/走廊）
* 选 connectors（接入连边）
* 分配流量

这是 strategic reusable-route network design（战略层可复用航路网络设计），不是 terminal layout design（终端布局设计）或 scheduler design（调度器设计）。交接总述和实验执行说明都明确把对象固定在“完整候选网络 + port-augmented graph（端口扩展图）”上。

## 4.2 中间层：终端如何进入网络

你们当前冻结的方法链是：

[
\text{raw factors} \rightarrow \text{canonical factor families} \rightarrow \text{intermediate states} \rightarrow \text{A/S/F} \rightarrow \text{network design}
]

更具体地：

* canonical factor families（规范因素族）：

  * (P)：procedure/interface regime（程序/接口体制）
  * (G)：geometry（几何）
  * (R)：resource configuration（资源配置）
  * (O)：organization/control（组织/控制）
  * (X)：context（上下文）
* intermediate states（中间状态）：

  * (H)：interface geometry state（接口几何状态）
  * (Q)：service/queue state（服务/队列状态）
  * (B)：boundary/footprint state（边界/脚印状态）
* network-facing interface（面向网络的接口）：

  * (A)：admissibility（可接性）
  * (S)：service capability（服务能力）
  * (F)：footprint/conflict（脚印/冲突）

## 4.3 A/S/F 各自的 adopted truth mechanism（采用的真值机制）

### A：几何与规则到接入可行性

A 这条线最硬，也最直观。
它来自：

* port 朝向
* sector 宽度
* regime rules
* context 下的禁止方向/禁止 connector

所以 A 最终是一个 admissibility object（可接性对象），决定“某个 connector 能不能接到某个 port”。这条链是几何/规则驱动的，理论上最容易写清。实验上由 EXP-1 打最小反例。

### S：资源竞争与排队近似到 port-level service

S 这条线不能只写成“一个二次式”，那样会显得拍脑袋。
正确写法应该分成两层：

**资源层（resource layer）**：
terminal 内部定义若干会竞争的资源：

* pad / gate
* merge point
* waiting ring / holding segment
* release slot / departure window
* terminal-wide control capacity

每个 port 的流量占用这些资源；共享资源会产生 cross-port drag（跨端口拖慢），全局共享瓶颈会产生 overall saturation（整体饱和）。
这层的理论来源就是 queueing / congestion / stochastic ATM literature（排队/拥堵/随机空管文献）。Shone 等综述、Peterson 1995、Itoh 2019 都说明机场和终端拥堵常常以排队/拥堵函数方式进入更高层的规划或控制问题。([科学直接][3])

**降阶层（reduced form）**：
由于这篇论文研究的是 network design，而不是 terminal operations（终端运行），所以不上 full microsimulation（全细节微观仿真），而是在工作区间把资源竞争压成 port-level convex cost + cross-port coupling（端口级凸成本 + 端口间耦合）。
这就是你们 truth service model 里 (a_h,b_h,m_{hh'},\bar\mu,\psi) 这一套。
在战略网络设计层，这样做是有充分先例的：枢纽/网络设计文献长期使用 congestion/capacity/service-level surrogates（拥堵/容量/服务等级代理），而不是 minute-level microsimulation（分钟级微观仿真）。([科学直接][2])

### F：局部空域构造与暴露外部性到 local edge penalty/block

F 是目前最脆、但也是最有潜力的那条线。
它不该被写成“一个凭感觉加的 penalty（惩罚项）”，而应该被写成：

* protected airspace / geofence / no-fly zone（受保护空域 / 围栏 / 禁飞区）
* airport procedure exclusion（机场程序排斥）
* terminal control area / waiting-ring envelope（终端控制区 / 等待环包络）
* local conflict exposure（局部冲突暴露）

这类 **local airspace constructs（局部空域构造）** 决定 terminal 周边哪些边被 block、哪些边基础上更贵。
理论与文献支撑来自：

* geofencing / keep-out volumes in low-altitude path planning（低空路径规划里的围栏与禁入体积）([MDPI][5])
* geometric airspace assessment / airport controlled airspace constructs（几何空域评估 / 机场受控空域构造）([MIT DSpace][6])
* terminal-area control / holding circles / multi-ring structures（终端区控制 / 等待环 / 多环结构）([科学直接][7])

但是要注意：
**F 没有 S 那样统一、成熟、标准的总公式。**
因此主文中 F 最稳的写法是：

* hard blocks（硬阻断）
* nominal local penalties（名义局部惩罚）

而不要把 load-sensitive footprint（随流量变化的脚印）写成主文的理论主轴。因为目前 4C 已经显示，这一项在近似求解下会带来 McCormick artifact（McCormick 伪影），容易把讨论拖偏。

## 4.4 四个模型各自是什么

实验统一比较四层：

* **M0：S-only**
  terminal 被压成粗节点，只暴露 aggregate service（聚合服务）
* **M1：AS**
  展开 ports 与 admissible connectors，保留 port-level service
* **M2：ASF**
  在 M1 上加入 local footprint / blocked-edge information
* **M***：truth benchmark
  保留更丰富的 terminal truth model（端口耦合、局部脚印、情景依赖）

## 4.5 评估协议：这篇论文最“硬”的地方之一

这篇不是比“哪个模型拟合得像不像”，而是比“哪个模型设计出来的网络在真值下更对”。

统一协议是：

1. 所有模型共享同一个 candidate graph（候选图）；
2. 各自基于自己的接口层做网络设计；
3. 固定设计以后，回到同一个 truth model（真值模型）下重算流分配；
4. 比较 truth-evaluated objective / regret / topology distance。

这条协议是整篇论文非常关键的贡献之一，因为它直接评估**设计误导（design misguidance）**，而不是模型拟合误差。

# 五、实验链：每个实验到底在证明什么

## 5.1 EXP-0：Factor Screening（因素筛查）

作用：
不是发现 ASF，而是检查 adopted interface hypothesis（采用的接口假设）在单 terminal 层面有没有非平凡响应。当前做法是：

* 600 个 base terminals（基础终端样本）
* 5 个粗粒度 factor families：`portGeom / routing / opConfig / context / demand`
* 单 terminal、无网络图、只看 (d_A,d_S,d_F) 响应。

当前结果说明：

* 三条通道都不是死的；
* routing 对 S 响应最强；
* context 对 F 响应最强；
* 但这仍是**粗 screening**，不能支撑“factor→channel 已被严密证明”。

所以在论文里，EXP-0 只能作为**方法支撑**，不能作为 headline result（主结果）。

## 5.2 EXP-1：A 的必要性

用 hand-built minimal counterexample（手工最小反例）证明：

* M0 会因为看不见 admissibility 选错 connector；
* 在 hard-cut 下甚至会选到 truth 下不可行的连接；
* M1 则能恢复 benchmark。
  结果非常硬：hard-cut 下 M0 在 truth 下直接 infeasible；soft 版本下 M0 的相对 regret 最高到 331.6%，M1 为 0。

## 5.3 EXP-2：S 的必要性

在 A 相同、图相同的最小图上证明：

* 即使两边都能接，只用 aggregate service 仍会误导；
* 低负荷时 M0 不犯错；
* ρ≥0.6 后，M0 regret 从 3.9% 一路升到 14.7%，最大 23.6%，M1 recovery rate 为 100%。

这是 S 的 strongest existence result（最强存在性结果）。

## 5.4 EXP-3：F 的必要性

在 A、S 都固定时证明：

* local footprint 足以独立改变网络拓扑；
* hard-block 下 M1 catastrophic failure（灾难性失败），M2 恢复；
* 这说明 AS 不一定够，某些场景必须 ASF。
  当前结果里，φ_F=1.0 的 hard-block 区域，M1/M0 都爆炸，而 M2 恢复为 0 regret。

## 5.5 旧版 EXP-4：第一次全网扫

它原本想直接画三段式适用区间图，但第一次 medium sweep 跑出来：

* M0 sufficiency 60%
* M1 sufficiency 59%
* M2 sufficiency 91%

于是暴露出两个大问题：

* M1≈M0
* ~10% baseline regret（基线误差）
  这个旧版结果现在已经不该当主文核心，而应作为“为什么要重构 4A/4B/4C/4D”的问题诊断材料。

## 5.6 EXP-4A：Calibration Gate（校准门控）

它不是科学命题实验，而是数值门禁。
目的：确认在 φ_F=0 子集上，M1/M2 的 baseline regret 已足够小，说明后面的 regime 图不是由 PWL 近似误差画出来的。
当前结果已经通过门控：在 φ_F=0 的 41 个实例里，M1/M2 的 median 和 P95 regret 都是 0，最大也只有 0.09%。

## 5.7 EXP-4B：A/S Isolation（A/S 通道隔离）

当前实际打出来的主要是 A，不是 S。

做法：固定 φ_F=0，只扫 ρ、α_A、κ_S，看 M0→M1 的升级价值 (U_{01})。
结果：

* 44% 的实例 U01 > 3%
* 当 α_A≈0.70 或 E_A≥0.7 时，中位 U01 跳到 14.8%
* κ_S 维度几乎没有形成清晰边界。

所以 4B 在当前稿子里最适合作为**A 的中型边界证据**，而不是“S 的完整分区图”。

## 5.8 EXP-4C：F Isolation（F 通道隔离）

固定 α_A=0.25、κ_S=2，只扫 ρ、φ_F，看 M1→M2 的升级价值 (U_{12})。
结果非常清楚：

* φ_F=0.5 时，U12 = +8.8%
* M0/M1/M2 的中位 regret 分别是 2083.3%、9.6%、0.6%
* 主边界大约在 φ_F≈0.5。

但这一步也暴露出一个问题：
φ_F=0.1~0.3 时 U12 为负，主要是 McCormick 松弛带来的保守伪影，不该被解读成“F 信息无价值”。这点在论文里必须正面讨论。

## 5.9 EXP-4D：Mixed Regime Map（混合适用区间图）

这是当前全网主实验。
它在 4 个图族、4 个 seeds、864 个完整网络实例上重新回答：

> 什么条件下推荐 M0，什么条件下推荐 M1，什么条件下推荐 M2？

当前结果：

* 推荐模型占比：M0 46.3%，M1 38.4%，M2 15.3%
* sufficiency：M0 38.3%，M1 72.3%，M2 67.9%
* φ_F=0 时 M1 被推荐 57%
* φ_F=0.5 时 M2 被推荐 45%。

这里有两个关键信号：

1. 旧版 `M1≈M0` 的大问题已经被实质修复；
2. 现在更适合把 4D 叫做 **hierarchical recommendation map（分层推荐图）**，而不是“完美无瑕疵三段式定律图”。因为中等 φ_F 区域仍有近似伪影、S 的独立边界也还没完全立住。

## 5.10 EXP-5：Realistic Proxy（现实代理）

固定一个更大的机场邻近图（6 terminals，4 waypoints，6 个情景），看现实感更强时网络图如何改变。
当前 strongest case 是 `high_constrained`：

* M0 regret ≈ 972.5%
* M1 regret ≈ 972.6%
* M2 regret = 52.1%
  说明 constrained airport-adjacent 情形下，footprint-aware design（脚印感知设计）对网络拓扑有巨大影响。

但要老实写限制：
当前这一个 proxy 里 `M0≈M1`，所以 EXP-5 主要支持 **F-heavy realistic case（F 主导的现实代理案例）**，不支持 A/S 的一般现实规律。

# 六、这篇论文当前最稳的主张，和不能写的主张

## 6.1 现在可以稳稳写的

1. **terminal 不能再被当成普通点节点。**
   这是 EXP-1/2/3 共同支撑的。

2. **A、S、F 三条通道都各自可能独立误导网络设计。**
   这是最小反例链支撑的。

3. **在 adopted truth class 中，A 和 F 的模型升级边界已经能被识别。**
   A 的中型边界大致在 `E_A≈0.7`；F 的中型边界大致在 `φ_F≈0.5`。

4. **统一的 design-then-evaluate-under-truth 评估协议是有价值的。**
   它直接评估网络设计有没有被误导，而不是接口拟合误差。

5. **在机场邻近、约束强的现实代理里，忽略 footprint 会灾难性改坏网络。** 

## 6.2 现在不要写死的

1. “ASF 是全局最小充分接口”
2. “S 的独立 medium-scale 边界已经完全清楚”
3. “EXP-0 发现了 ASF”
4. “truth class 就是现实终端机理的唯一正确形式”

这些话都会把稿子带到不必要的风险上。

## 第 1 章：引言（Introduction）

### 这一章要完成什么

把问题从“低空/UAM 很热”推进到：

> **战略网络设计现在缺的不是更多 terminal 运行细节，而是一个能被网络设计直接调用的终端接口层。**

### 要写的逻辑

1. 低空/UAM 正从点状示范走向 network design（网络设计）阶段。
2. 现有网络设计通常把 terminal 当成同质节点。
3. 现实 terminal 明显异质，尤其体现在：

   * 接入规则与可行方向
   * port 级服务能力
   * 终端邻域的局部空域约束
4. 于是问题变成：**为了把网络设计做对，终端需要暴露什么信息？**
5. 本文贡献不是更大一体化模型，而是：

   * 一个分层终端接口假设 A/S/F
   * 一个 truth-evaluated design protocol（真值重算设计协议）
   * 一条从存在性到适用区间的证据链
   * 一个现实代理案例

## 第 2 章：相关研究与本文位置（Literature and positioning）

### 这一章要完成什么

告诉读者：你们不是在重复 terminal operations，也不是在重复 corridor design，而是在二者之间补一层 missing interface（缺失接口）。

### 建议分成四节

#### 2.1 战略网络设计 / 枢纽网络设计

要说清：这类文献已经允许 congestion/capacity/service-level surrogates（拥堵/容量/服务等级代理）进入网络设计，所以你们在战略层用中观代理不是越界。([科学直接][2])

#### 2.2 空管 / 机场拥堵与排队

说明 S 的理论来源：queueing/congestion/resource competition（排队/拥堵/资源竞争）。
引用 Shone 综述和机场 queueing 相关工作。([科学直接][3])

#### 2.3 UAM / 低空空域设计与几何空域构造

说明 F 的来源：airspace constructs、airport procedures、protected zones、geometry。([科学直接][4])

#### 2.4 geofencing / terminal-area control / vertiport operations

说明 local keep-out / holding circles / multi-ring / terminal-area rules 这些对象确实存在于文献里，但没有被系统压成 network-facing interface。([MDPI][5])

### 本章结尾一定要写的定位句

> **本文不是终端运行设计论文，也不是新算法论文；本文是一篇 network-design interface paper。**

#### 4.3 S 的 adopted mechanism

## 第 10 章：讨论（Discussion）

这章很关键，必须正面处理边界。

### 要明确写的四件事

1. **本文证明了什么**
   terminal 不是点；A/S/F 各自会误导网络；A 和 F 的升级边界已识别；机场邻近时 footprint-aware design 会改网。

2. **本文没有证明什么**
   没有证明 ASF 是全局最小充分接口；
   没有证明 S 的独立 medium-scale 边界已经完全锁死；
   没有证明 truth class 是现实终端运行的唯一正确模型。

3. **EXP-0 的真实地位**
   screening / sanity check，不是 discovery module。

4. **F 的方法边界**
   主文 F 是“局部空域构造与几何暴露外部性”的 adopted class，不是完整的 terminal-conflict science（终端冲突科学）。

---

## 第 11 章：结论（Conclusion）

结论不该写成：

> “我们证明了 ASF 必要。”

最稳的写法是：

> 对低空可复用航路网络设计而言，异质终端不能再被视为点节点；
> 一个分层终端接口可以把 terminal 的关键异质性压缩成对网络真正有用的设计信息；
> 在采用的真值类中，A 和 F 分别对应明确的模型升级边界，而终端区内部复杂性不必整体抬升到上层联合优化。

# 八、当前稿件最关键的风险与待办

## 8.1 当前最弱的一环

**EXP-0 仍然是最弱的一环。**
如果不重构，它只能支持“ASF 不是空接口”，不能支持“factor→channel 已被严密建立”。所以写作时一定要降格使用。

## 8.2 当前最强的一环

**统一评估协议 + EXP-1/2/3 + 4A/4B/4C/4D 的证据链**
这是整篇最硬的部分，应该成为摘要和贡献点里的核心。

## 8.3 最好补的动作

1. 统一所有网络图可视化数据输出（节点、ports、扇区、被选边、被 block 边、truth flow）
